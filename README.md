## Architecture

- **Blazor Interactive Server** frontend with SignalR hubs for real-time updates
- **ASP.NET Core API** for admin and app endpoints
- **PostgreSQL** for persistence
- **Twitch OAuth** for authentication

## Prerequisites

- .NET SDK **10.x**
- Docker + Docker Compose (for containerized setup)

## First-Time Setup

1. Copy `.env.example` to `.env`.
2. Set required values:
  - `DB_PASSWORD`
  - `TWITCH_CLIENT_ID`
  - `TWITCH_CLIENT_SECRET`
  - `ADMIN_API_KEY`
3. Optional telemetry values (Azure Application Insights):
  - `APPLICATIONINSIGHTS_CONNECTION_STRING` (preferred)
  - `APPINSIGHTS_INSTRUMENTATIONKEY` (legacy fallback)
3. Create local mount-point folders if they do not already exist:
  - `mounts/postgres-data`
  - `mounts/app-logs`
  - `mounts/dev/postgres-data`
  - `mounts/dev/pgadmin-data`
  - `mounts/dev/nuget-cache`
  - `mounts/dev/seq-data`
  - `mounts/dev/redis-data`

## Local Development

Run from the repository root:

```bash
dotnet run --project src/PS2Challenge.Main/PS2Challenge.Main.csproj
```

```bash
dotnet build src/PS2Challenge.sln -c Release
```

## Docker

### Configure environment

Copy `.env.example` to `.env` and set at least:

- `DB_PASSWORD`
- `TWITCH_CLIENT_ID`
- `TWITCH_CLIENT_SECRET`
- `ADMIN_API_KEY`
- `APPLICATIONINSIGHTS_CONNECTION_STRING` (optional)

The app resolves the database connection string in this order:

- Azure App Service connection-string variables (`POSTGRESQLCONNSTR_DefaultConnection`, then any `POSTGRESQLCONNSTR_*`)
- `DATABASE_CONNECTION_STRING` (recommended fallback for Docker/local environment variables)
- `ConnectionStrings:DefaultConnection` (for `appsettings*.json` or `ConnectionStrings__DefaultConnection`)

Twitch values are loaded from `TWITCH_CLIENT_ID` and `TWITCH_CLIENT_SECRET` (with `Twitch:ClientId` / `Twitch:ClientSecret` as configuration fallback). If `APPLICATIONINSIGHTS_CONNECTION_STRING` (or legacy `APPINSIGHTS_INSTRUMENTATIONKEY`) is present, Application Insights telemetry is enabled automatically.

### Production-style compose

```bash
docker compose -f docker/docker-compose.yml up -d
```

- App: `http://localhost:5001`
- Postgres data mount: `./mounts/postgres-data`
- App logs mount: `./mounts/app-logs`

### Development compose

```bash
docker compose -f docker/docker-compose.dev.yml up -d
```

- Uses `dotnet watch` for the app service
- Mounts source from `./src`
- Includes optional tooling/services: pgAdmin, Seq, Redis
- Development mounts are under `./mounts/dev/*`

## CI/CD

- **CI Build and Test** (`build.yml`)
  - Runs on `pull_request`, `push` to `main`, and manual dispatch
  - Builds once, publishes, and runs all test suites
- **CI Deploy** (`deploy.yml`)
  - Runs only when CI Build and Test succeeds for a `push` on `main`
  - Deploys the publish artifact generated by CI (no rebuild during deploy)

## Project Layout Notes

- Docker-related runtime files are under `docker/`:
  - `docker/Dockerfile`
  - `docker/docker-compose.yml`
  - `docker/docker-compose.dev.yml`
- `.dockerignore` remains at repository root because Docker applies ignore rules from the build context root.

## Troubleshooting

- Check `/health` for container/application health checks
- Ensure PostgreSQL is reachable from the app container (`Host=postgres;Port=5432` in compose)
- Verify Twitch credentials are set when running non-test environments
